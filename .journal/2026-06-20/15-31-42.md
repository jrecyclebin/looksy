## Session Update

### Completed

- Overhauled `processOutput`: no longer extracts XML blocks — prints full LLM response verbatim, then appends expanded file contents after a `---` separator
- Made reference detection robust: `refRe` now matches refs anywhere in a line (not just at start), supports single line numbers (`main.go:77`) in addition to ranges (`main.go:77-90`)
- Added `extractComment`: finds ` — ` or ` - ` after a ref match (skipping markdown noise) and captures the description
- Updated `expandReference` to accept `refWithComment` string, splitting ref from comment at first space
- Added `actualPrompt` variable for blockquoting the user query so the LLM knows it's research material, not instructions to execute
- Simplified `promptPostamble`: dropped `<summary>`/`<references>` XML tag instructions, now just asks for refs with examples
- Removed `extractBlock` (dead code after the output overhaul)
- Added `exec.LookPath` check for the search tool — falls back to `grep` with a warning if `rg` (or whatever was chosen) isn't on PATH
- Updated README: fixed stale example output, filled in the Configuration section (was a stub calling itself "Jabsco"), corrected supported tool list
- 16 tests passing

### Key Decisions

- **Pass-through + append**: Instead of trying to parse the LLM's response structure (which varied wildly between models), just print it as-is and tack on expanded refs. Eliminates the whole "LLM didn't use XML tags" problem.
- **Dash-delimited comments**: Rather than trying to strip markdown formatting after the ref, look for ` — ` or ` - ` as the comment delimiter. This is the pattern the prompt already suggests, and it skips bold/backtick noise naturally.
- **`exec.LookPath` for search tool**: Simple and reliable — if `rg` isn't installed, we don't want the LLM trying ripgrep commands that won't work.

- The old `TestRefLineRe` test name was stale after renaming `refLineRe` to `refRe` — updated along with the test cases
